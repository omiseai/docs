# NFG メニュープロトコル（日本語訳・共有用）

> この文書は、実際にエージェントが実行する `SKILL.md`（英語）の**人間向け日本語訳**です。
> 内容の確認・共有のためのものであり、エージェントが読み込むのは英語版です。
> 両者に食い違いがあれば英語版が正です。

## なぜこれが必要か

OmiseAI のコンシェルジュは、アレルギーに関する質問を含め、お客様の質問に **NFG
データセットをもとに**回答します。データが「この料理はピーナッツ不使用」と言って
いて実際には入っていれば、人が傷つきます。だからこの作業の目的は「メニューを説明
すること」ではありません。**安全に関わるすべての記述が、信頼できる出典まで辿れる**
こと、そして**分からないことは『分からない』とはっきり見える形で残す**ことです。
もっともらしい推測で穴を埋めてはいけません。

どれほど上手にプロンプトを書いても、AI は時々、材料・アレルゲン・酒のスペックを
でっちあげます。プロンプトの工夫だけでは防げません。そこで、でっちあげを**構造的に
起こりにくく**します ——「観察した事実」と「推測」を分け、アレルゲンは材料から機械的
に導き、検証済みの正規データで AI の出力を上書きし、空白は事実になる前に必ず人へ
回す。このプロトコルはその仕組みを、手作業で実行できるよう書き出したものです（将来
は自動エージェント化します）。

指針（NFG の設計思想より）: ハルシネーション（過剰）と空欄（不足）の中間にあるのが、
**検証され、信頼度を明示したデータ**です。**見せかけの完全さより、正直な不完全さ**を。

## 1回の実行で作られるもの

1回の実行は**1店舗のメニュー全体**を扱い、店舗ごとのフォルダに番号付きの
Markdown ファイル群を書き出します:

```
nfg-runs/<店舗スラッグ>/
├── 01_source_capture.md         受け取った入力すべての忠実な書き起こし（解釈なし）
├── 02_decomposition.md          料理 → 材料＋調理法、全項目に出典タグ
├── 03_allergens_restrictions.md 材料から導いたアレルゲン・制限（導出の連鎖を明示）
├── 04_gap_register.md           オーナーへの確認事項、安全性を最優先に並べる
├── 05_operator_dialogue.md      確認事項のやりとり（その場 and/or 非同期）
├── 06_nfg_output.md             NFG モデルに沿ったデータセット —— まず下書き、後で確定
└── 07_operator_review.md        まとめた内容をオーナーに見せて承認をもらう（VAD）
```

番号は処理の順番です。先回りしてはいけません —— 各段階は前段階が正直であることに
依存します。**オーナーとの接点は2回あり、性質が異なります**。05 は特定の空白を埋める
ための個別質問。07 は**組み上げた全体像**を見せ、特に推測しかできなかった箇所を
確認・修正してもらう段階です。`06_nfg_output.md` が「出力」（DB に保存するものに相当）
で、最初は**下書き**、07 の承認後に初めて確定します。

実行の冒頭で自分でフォルダを作ります。店名をスラッグ化します（例「Bonta 西梅田店」→
`bonta-nishi-umeda`）。1品だけ試す場合も全工程を回せます（各段階が縮小されるだけ）。

## 出典ルーブリック —— このスキルの核心

料理に関するすべての事実（材料・調理法・アレルゲン・スペック）に、**出典タグ**と
**信頼度**を付けます。これがハルシネーションをお客様に届かせない仕組みです。
タグは `… ピーナッツ [INFERRED·low]` のように記述に直接付けます。

| タグ | 意味 | 信頼度 |
|---|---|---|
| `[MENU]` | メニュー／提供素材に記載あり | 高 |
| `[OPERATOR]` | オーナー本人が確認（これが VAD） | 最高 |
| `[CANONICAL]` | 検証済み資産（product_master / ingredient_glossary / 食文化KB）に一致 | 最高 |
| `[INFERRED]` | 料理の常識からの AI の推測 —— **未検証** | 低 |
| `[UNKNOWN]` | 本当に不明 | なし |

信頼度は `high` / `med` / `low`。出典の中でどれだけ確かかを表します（例: ぼやけた
メニュー写真なら `[MENU·med]`）。

**全体を安全にする鉄則:**

> アレルゲン・食事制限・宗教上の制限に関わる `[INFERRED]` / `[UNKNOWN]` の記述は、
> **絶対に**最終データセットに事実として書いてはいけません。代わりに確認事項リストの
> 質問にします。推測は説明文（ナラティブ）には使ってよいが、安全項目には禁止です。

例:「とんかつは衣がついているので小麦・卵を含む可能性が高い」は妥当な推測 —— ただし
衣と揚げ油をオーナーが確認するまで、小麦／卵／大豆の有無は**確認事項**であって、
データの事実ではありません。安全に関わるか迷ったら、関わるものとして扱います。

## 正規（カノニカル）資産 —— どこを見るか、無い場合どうするか

`[CANONICAL]` タグは、本当に検証済みのデータを指すときだけ正直なタグです。
このスキルは正規資産（とスキーマ）を `assets/canonical/` に同梱しています:

- `product_master.json` —— 検証済みの飲料・製品スペック（日本酒・焼酎・ワイン…）
- `ingredient_glossary.json` —— 検証済みの英語材料名＋境界ルール
- `food_culture_kb.json` —— 検証済みの郷土料理の文化的説明

実行冒頭でこれらを探します。本番では別パスや DB から提供されることもあります ——
より充実したコピーをユーザーが指したら、そちらを優先します。照合ルール（龍 ⊂ 黒龍 の
部分一致の罠を含む）は `assets/canonical/README.md` を参照。

**資産が無い／どの項目にも一致しない場合 —— ここが重要です:** 穴を埋めるために
正規データを**でっちあげてはいけません**。捏造した「正規」データは最悪の失敗です。
推測を検証済みのように見せかけ、しかもすべてを上書きしてしまうからです。代わりに:

1. 一致しない事実は `[INFERRED]`（または `[UNKNOWN]`）として扱い、通常の安全ルールを
   適用 —— アレルゲンに関わるなら確認事項にする。
2. **候補（candidate）**として、最善の下書きを
   `assets/canonical/candidates/<asset>.candidates.jsonl` に `status: "draft-unverified"`
   付きで追記してよい。候補は**正規ではなく**、人が（蔵元スペック・用語集・MAFF 等で）
   検証して昇格させるための待ち行列です。「AI が下書き → 人が差分をレビュー → 適用」
   の原則そのものです。

自分が書いた候補を、同じ実行内で `[CANONICAL]` として読み戻してはいけません。

## 各段階

各ファイルの正確な書式は `references/stage-templates.md`、対応づける列挙
（8つのマスター、32項目・5か国のアレルゲン表、12の制限、調理法、正規資産ポリシー）は
`references/nfg-data-model.md` を参照。必要になった段階で読み込みます。

### 段階 01 —— ソース取り込み（`01_source_capture.md`）
受け取ったものすべてを、原語のまま、解釈を加えず書き起こす。写真・PDF・表・貼り付け
テキストすべて。各品目の表記名・説明文・価格・区分を記録。各ソースの媒体と品質も
メモ（「写真、ややぼやけ、左下が読みにくい」）。本当に読めない箇所は `[illegible]` と
書き、推測しない。これは「実際に受け取ったもの」の基準点で、後段はこれと照合されます。

### 段階 02 —— 分解（`02_decomposition.md`）
各料理を構成要素 → 材料へ分解し、NFG の材料モデル（単純 / 複合（`components` 付き）/
加工食品）に対応づける。カレールウのような複合材料は 小麦＋油脂＋牛肉＋乳 に展開する
—— アレルゲンは複合材料の中に隠れるため。調理法は20種から割り当てる。**全項目に**
出典＋信頼度を付ける。ここで正規資産を適用 —— `product_master` や `ingredient_glossary`
に一致すれば `[CANONICAL]` とし、その値を使う（推測を上書きする）。用語集の境界ルール
に従う（英語名があればそれ、ローマ字＋注釈は日本固有のものだけ）。ソースでなく常識で
埋めたものは `[INFERRED]`、欠けているものは `[UNKNOWN]`。

### 段階 03 —— アレルゲン・制限（`03_allergens_restrictions.md`）
アレルゲンは**材料から導く。料理に直接タグ付けしない**。各料理について材料リスト
（展開した複合材料の構成要素を含む）をたどり、導出の連鎖を示す —— どの材料がどの
アレルゲンを示すか、32項目・5か国対応で。12の食事/宗教制限も同様に。導いた
アレルゲンの出典は、連鎖の中で**もっとも弱い**ものを継ぐ —— 衣が `[INFERRED]` なら
小麦アレルゲンも `[INFERRED]`、つまり事実ではなく確認事項になる。苦労して得た区別を
守る: ゼラチンは独立トークンで肉アレルゲンではない、ポーク不使用/ハラルはアレルゲン
ではなく制限の層、デザートに肉が誤タグされる類の誤りに注意。確認事項の多くはここで
生まれます。

### 段階 04 —— 確認事項リスト（`04_gap_register.md`）
すべての `[UNKNOWN]` と、安全に関わる `[INFERRED]` を、オーナーへの番号付き質問に
まとめる。**安全性を最優先に並べる**: アレルゲン・制限に関わるものを最上位、次に材料の
同定、最後に説明・文化的な空白。各質問は、料理・具体的な不明点・なぜ重要か・（有効なら）
忙しいオーナーが答えやすい選択肢形式を示す（「天ぷらの揚げ油は a) 純植物油 b) ごま油
入りブレンド c) その他？」）。良い確認事項こそが B 評価と S 評価の差です。

### 段階 05 —— オーナーとのやりとり（`05_operator_dialogue.md`）
確認のやりとり。2通りで行い、両方ここに記録:

- **その場（ライブ）**: 対話的に実行しているときは、（オーナー役の）ユーザーに最優先の
  安全質問を会話の中で直接尋ねる。リストを全部ぶつけず、安全質問を先に少数ずつ。
  回答は `[OPERATOR]` として扱う。
- **非同期**: `04_gap_register.md` 自体が、後でオーナーが答えられる成果物。回答が来たら
  日付と回答者とともにここに記録。

「品によって違う」という答えを個々の料理に波及させない —— オーナーが揚げ油を「料理に
よる」と言っても、特定の料理の油は確定しない。料理ごとに空白を開けたままにする。
特定の項目が確定したときだけ `[OPERATOR]` として記録する。

### 段階 06 —— NFG 出力の下書き＋検証ゲート（`06_nfg_output.md`）
NFG モデルの形でデータセットを組む: 料理ごとに、機械的意味（ingredient_ids,
method_ids, 導出アレルゲン, 制限, 味プロファイル, カロリー帯）と、人間向けナラティブ
（物語, 文化的背景, 提供メモ）。段階05の回答を反映し、オーナーが確認した箇所の出典を
格上げし、料理ごとに暫定の **`verification_rank`** を付ける:

- **S** —— 安全に関わる全項目が `[OPERATOR]` か `[CANONICAL]`、完全。
- **A** —— 主要項目は確認済み、軽微なナラティブの空白のみ。
- **B** —— 重要項目に `[INFERRED]` が残る、または安全の空白が残る。
- **C** —— ほぼ `[INFERRED]`、オーナー確認前の下書き。

ファイル末尾に**検証ゲート**チェックリスト（テンプレ参照）。次を満たさないと「合格」
しない: 安全項目が `[INFERRED]`/`[UNKNOWN]` に依存していない、各アレルゲンが材料まで
辿れる、正規の上書きを適用済み、機械項目とナラティブを分離、残る不完全さを評価とともに
明示。S/A に届かなくても問題ない —— 空白を明示して B/C で出す。**見せかけの完全さより
正直な不完全さ**。

段階07 までは**下書き**。ヘッダに `status: draft (pending operator sign-off)` と記す。
承認後に戻って確定する: オーナーの修正を反映、確認項目を `[OPERATOR]` に格上げ、評価を
再計算、ヘッダを `status: finalized (operator-signed YYYY-MM-DD)` に変更。

### 段階 07 —— オーナー確認・承認（`07_operator_review.md`）
2回目のオーナー接点であり、VAD が実際に得られる瞬間です。オーナーは読めて理解できる
ものしか確認できないので、組み上げた**全体像**を平易な言葉で、**オーナーの言語で**
見せます（対象の和食店なら日本語で書く。英語の対訳はユーザーが求めたときだけ）。
狙いは、特に推測しかできなかった箇所について、判断する機会を本当に与えることです。

忙しいオーナーが動きやすいよう構成する:

- **各料理について我々が考えていること**を平易に —— 材料、調理法、どのアレルゲンを
  含むか、誰が食べられて誰が食べられないか。
- **すべての `[INFERRED]` 項目を**「我々の推測 —— 要確認」として明示。オーナーが
  もっとも見るべき行。`[INFERRED·high]` でも推測は推測で、真実にできるのは
  オーナーだけ。各項目に簡単な操作を: ✔ そのとおり / ✖ 違う → 正しくは…。
- **すでに確かなもの**（`[MENU]` `[CANONICAL]` `[OPERATOR]`）は別に示し、確認不要な
  ものを再確認させない —— 時間を尊重する。
- 安全なアレルゲン回答を**まだ妨げている未回答**の確認事項を挙げ、現在の
  `verification_rank` の意味を一文で（「この水準ではコンシェルジュは X について
  『スタッフにご確認ください』と案内します」）。

オーナーが回答したら、確認・修正をここに（日付・回答者付きで）`[OPERATOR]` として
記録し、段階06 に戻って確定する。オーナーが我々の推測を下方修正する（「唐揚げは小麦
でなく片栗粉です」）のは、システムが意図通り働いている証拠 —— その修正がすべてに優先。

この承認までデータは下書きで、実行サマリーにもそう書く。完全に成功した実行の成果物は、
**オーナー承認済みで確定した**段階06 であり、残る安全の空白がゼロの状態です。

## 進め方

- **テンプレを埋めるために捏造しない**。`[UNKNOWN]` のタグ付き空欄は失敗ではなく成功。
  価値は確認事項リストにある。
- **結論を主張せず、導出を示す**。安全に関わるものは、読み手がアレルゲン → 材料 →
  出典 を辿れるように。
- **目視より検証コードを優先**。段階06 のアレルゲンがすべて段階03 の導出に出てくるか、
  `[INFERRED]` 出典のアレルゲンを持つ料理が無いか等は、読み直すより短いスクリプトの方が
  確実。
- **`data_source` に注意**。このプロトコルは `live` 店舗向け。コーパス（`user_id = 19`）
  データへの書き込みは絶対にしない。

実行を終えたらユーザーに要約する: 料理数、S/A に届いた数、まだ妨げている安全の空白の数
—— 最後の数字がもっとも重要です。
