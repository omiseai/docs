# Context Brief: Hermes Agent Multi-Business Deployment Project

## What this project is

Matt is building agents for businesses using **Hermes Agent** (NousResearch's open-source, self-hosted agent harness — MIT licensed, not affiliated with Anthropic). Each business gets its own deployment; employees at that business interact with it via LINE and Slack. The plan is to run this repeatedly across many businesses, so the setup needs to be a template, not a one-off.

Current target architecture (all decisions final unless noted "open" below):

- **One Render web service (Docker) per business**, using the [Hermes on Render template](https://render.com/templates/hermes-on-render) — gives business-to-business isolation at the container level.
- **Multiple Hermes "profiles" per business = "virtual employees."** Each virtual employee is a distinct persona/agent (own `SOUL.md`, own config, own LINE/Slack bot credentials) that all of that business's human employees can talk to.
- **Honcho Cloud** as the memory provider, one workspace per business, to solve multi-employee identity modeling (see below — this was the crux of a long back-and-forth).
- **Terminal/shell execution disabled entirely** for every virtual-employee profile — no employee, at any business, can get an agent to run shell commands.
- **Admin (Matt) access is via Render's own shell/SSH**, completely outside Hermes — not via any Hermes profile or chat interface.
- **Backups**: both settings-as-code (git-tracked `config.yaml`/`SOUL.md`/skills, doubling as the template for the next business via Hermes' "Profile Distributions" feature) and scheduled full-data snapshots (`hermes profile export`/`hermes backup`) to external storage.
- **Cross-virtual-employee collaboration** happens via shared Google Docs/Dropbox access, not shared memory — Matt's own idea, validated as sound.

## Key Hermes concepts established during this conversation

**Profiles vs. per-user access are orthogonal.** A "profile" (`hermes profile create <name>`) is a separate agent identity — own config, memory, sessions, skills, credentials, optionally own bot token. It is *not* the mechanism for giving multiple humans access to one agent. That's handled by the gateway's own authorization (allowlists or DM pairing) plus automatic per-user session isolation (`group_sessions_per_user: true`, the default) — one bot, one profile, many employees, each gets an isolated conversation automatically. Matt's earlier belief that "each employee needs their own profile" (formed from a Telegram experiment) was actually about wanting two separate bots, not a per-user requirement.

**`group_sessions_per_user`** controls whether a virtual employee's *group* chats give each participant a private thread (`true`, default) or become one shared conversation everyone contributes to (`false`). This is one on/off switch per profile — confirmed **not** to be settable per individual channel/group, and does **not** affect DMs at all (DMs are always private, keyed independently by chat ID regardless of this setting). Matt accepted this as all-or-nothing per virtual employee; if a business needs some groups shared and others private for "the same" virtual employee, that requires splitting into two separate profiles instead.

**Profiles are a state boundary, not a security boundary.** This was the single most important correction in the conversation. On the default `local` terminal backend, any profile has the same OS-level filesystem access as the account Hermes runs as — nothing stops one profile from reading another's memory/sessions/`.env` secrets except that no built-in *tool* does so by default. There is no OS-level (chmod) separation between profile directories; it's directory convention, not an enforced wall. Real isolation only comes from the *terminal backend* (Docker/Modal/Daytona), where "the container is the sandbox boundary" and dangerous-command approval is skipped because it's unnecessary. On Render specifically, the whole business's Hermes instance runs in one container — that gives business-to-business isolation, but not employee-to-employee or profile-to-profile isolation *within* that container.

**Why Docker-as-terminal-backend needs a "sibling" container:** when `terminal.backend: docker` is set, Hermes doesn't run shell commands in its own container — it spawns and drives a *separate*, persistent sandbox container via the Docker API, so a destructive/misdirected command can only damage that disposable sandbox's mirrored filesystem, not Hermes' real `HERMES_HOME`. Confirmed: the sandbox's `/root/.hermes` is bind-mounted to a sandbox-local mirror directory, genuinely distinct from the real profile directory; skills/credentials mount in read-only; host cwd isn't shared unless opted in. To create that sibling container, Hermes needs to reach a Docker daemon, which means bind-mounting the host's `/var/run/docker.sock` (Docker-out-of-Docker) — which grants near-root host privileges and is why a managed PaaS like Render is unlikely to expose it. **Decision made:** since terminal is disabled for everyone anyway (see guardrails below), this is moot for now, but Modal/Daytona (cloud sandbox APIs, no local socket needed) would be the fallback if sandboxed execution is ever needed later.

**Memory: built-in vs. Honcho, and why the USER.md model doesn't fit multi-employee use.** Hermes' built-in memory (`MEMORY.md`/`USER.md`) is a small (~500-800 token), manually-curated, single flat file per profile with **no concept of "who said this."** It assumes one ongoing human↔agent relationship. If multiple employees share one virtual employee's profile, the agent may write one employee's stated preference into the one shared `USER.md` and it'll silently apply to everyone who talks to that virtual employee — a real blending problem, not a hypothetical one. Session isolation (each employee gets their own conversation thread) does **not** fix this, because memory lives at the profile layer, not the session layer.

**Honcho** is the memory-provider plugin that actually solves this: it models conversations as "peers" (an identity, human or AI) exchanging messages, supports multiple peers under one shared "workspace," and does automatic LLM-driven reasoning to build a per-peer representation. It's one of 8 swappable memory providers Hermes ships (Mem0, Supermemory, Hindsight, etc.) — kept as an optional plugin rather than merged into core because it requires its own hosted/self-hosted backend and background LLM reasoning calls, which would be unwanted overhead for users who just want the lightweight single-user default.

**The critical Honcho config decision:** in the gateway, `userPeerAliases` must map each employee's platform runtime ID (LINE/Slack user ID) to their own distinct peer name. **`pinUserPeer: true` must NOT be used** — it collapses every human talking to a virtual employee into one shared peer identity, recreating the exact blending problem Honcho exists to solve. This is called out explicitly in the checklist (Phase 3) as the detail most likely to be gotten wrong.

**Templating across businesses:** Hermes' "Profile Distributions" feature (package a profile's SOUL/config/skills/cron/MCP connections as a git repo, `hermes profile install github.com/you/template --alias`) is the intended mechanism for stamping out the same virtual-employee setup at each new business — credentials, memories, and sessions stay per-install, so you centralize the *template*, not the data. Each business still needs its own Honcho workspace/API key so businesses' data never actually pools together, even when using the same template.

## Known gaps / open risks (not fully resolved — treat as still-open, not settled)

1. **`session_search` may not be scoped per user.** This tool lets an agent recall past conversations via full-text search across the profile's entire session database. The docs don't confirm it filters by which employee is asking, and there's an open GitHub issue proposing per-user session isolation/RBAC — suggesting this is a recognized, not-yet-closed gap. If an employee asks "what did my coworker tell you last week," it's untested whether the agent would (or should) be able to answer. **Action item:** test this directly on a real instance before trusting it with sensitive per-employee content. This is separate from Honcho — Honcho fixes the *memory/preference-modeling* blending problem, not this raw-transcript-search exposure.
2. **MCP tool calls (Google Docs, Dropbox, any third-party integration) have no approval gate today.** The dangerous-command detection, approval prompts, and hardline blocklist are wired specifically into the `terminal`/`execute_code` tools. A destructive action via an MCP-connected service (e.g., deleting a shared doc) would execute immediately with no human-in-the-loop check from Hermes — confirmed via multiple open feature requests asking for exactly this. Mitigation is happening outside Hermes' own systems: scope credentials to read-only/narrow permissions wherever possible, rely on the destination service's own recovery (Google Drive trash, Dropbox Rewind), and consider a custom gateway hook if write/delete scope is ever granted.
3. **Per-user/role-based access control within a shared profile doesn't exist yet.** Approval prompts for risky commands route back to whoever triggered them — there's no way today to have "employee requests, only admin approves." (Moot for now since terminal is disabled entirely, but relevant if that ever changes.)
4. **A known Docker-backend bug**: file writes from tools can silently land in the sandbox-mirror copy of profile state rather than the real file, with no error — worth testing directly before relying on `terminal.backend: docker` for anything, if it's ever reintroduced for a future admin profile.
