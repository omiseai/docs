# 質問生成エンジン 仕様書 (Question Engine)

最終更新: 2026-06-06 / 状態: ドラフト（実装中）

店主への「AI確認事項」を、チャネル（管理画面 / LINE）に依存せず一元生成するルールの正本。
LINEは配信路にすぎない。ここで質問の**中身と出し分け**を固める。

---

## 0. 設計思想

- 集めたいデータを聞くのではなく、**AIの不確実性を潰すのに必要な最小限**だけ聞く。
- 質問は「入力してください」ではなく「**AIはこう判断した。合ってる？**」（確認）を基本にする。
- 粒度は**店主が記憶から即答できる事実**の層で止める。そこから先（アレルゲン導出等）は店主に聞かず知識ベースが計算する。
- 安全項目は嘘の「無し」より正直な「**不明**」を優先する（誠 / NFGハルシネーション方針の "書かない" を安全に明示化）。
- 開発判断基準: **その機能は店主のタップ数を減らすか？** 減らさないものは優先度を下げる。
- **店主に出す全テキスト（質問本文・選択肢・ボタンラベル）は敬語（丁寧語）。ため口禁止。** 例: ○「合っていますか？」「わかりません」「含みます」 / ✗「合ってる？」「わからない」「含む」。

## 1. 優先度ティア（店主向け表示）

内部ランク（S/A/B/C）は店主に見せない。表示は3分類。

| ティア | 表示名 | 対象 | 保存先フィールド |
|---|---|---|---|
| `safety` | 公開前に確認 | アレルゲン / アルコール / 豚由来 / 牛由来 / 生食 / 宗教制約 | `allergens` / `restrictions` / `cooking_methods` / `is_alcoholic` |
| `order` | 注文判断に影響 | 辛さ / 量 / 子供向け / 限定品 / 提供可否 | `taste_values` / `serving` / `featured_tags` / `status` |
| `appeal` | 魅力アップ | ペアリング / 食べ方 / こだわり / ストーリー | `narrative`(pairing_suggestion / chef_note / story 等) |

- `safety` は**上限で切らない**（最優先・絶対に残す）。`order` / `appeal` のみ件数上限を適用。
- ソート: `safety` → `order` → `appeal`、各ティア内は優先度スコア降順。
- **飲料を一律スキップしない**。酒/ビール/ワインは度数・未成年・妊婦・宗教で `safety`（アルコール確認を出す）。ソフトドリンクや料理専用項目は適用範囲(`applies_to`: food/drink)で出し分け。

## 2. 質問形式の出し分け（核心）

低確信で1個だけ断定提示すると店主が「まあいいか」で誤承認する（acquiescence bias）。
**確信度の帯**で形式を変える。

| 条件 | 形式 | 例 |
|---|---|---|
| 安全項目・値あり（確信度問わず） | **確認型（編集可）** | 「アレルゲンは【小麦・卵】で合ってる？」※チェック増減できる |
| 安全項目・値なし | 列挙（断定しない） | 「このサラダのドレッシングは？」 |
| 非安全・確信度 ≥ 0.85 | **聞かない**（信頼） | — |
| 非安全・0.55 ≤ 確信度 < 0.85（惜しい） | **確認型** | 「辛さは"中辛"で合ってる？ 1.正しい 2.直す」 |
| 非安全・確信度 < 0.55（割れてる） | 列挙（断定しない） | 候補から選ぶ |

閾値定数: `CONFIDENCE_CONFIRM_HIGH = 0.85` / `CONFIDENCE_CONFIRM_LOW = 0.55`（要承認・後で調整可）。

確認型の選択肢には常に**共通の出口**を付ける（エンジンが質問オブジェクトの `actions` として必ず出力。アダプタ任せにしない）:
1. 合っている / 2. 違う（直す）→ その場で列挙にフォールバック / 3. 該当なし（例: この料理に出汁は使わない）/ 4. わからない（§4 unknown終端へ）
- 列挙(select/multi)は「該当なし / わからない」、item_confirm は「合っている / 修正する」。

## 3. 深さの3段エスカレーション（できるだけ浅く止める）

| 段 | 何を聞く | 例 | 終端 |
|---|---|---|---|
| L1 店舗共通 | 使ってる製品・方法（既存 `kitchen_profile.py`） | 「メインのドレッシングは？」 | CASCADE で8割確定 |
| L2 個別の例外 | その料理だけ違う時（確認型1タップ） | 「このサラダだけ自家製？」 | 確定 |
| L3 自家製/varies/不明 | 中身の**材料**（アレルゲン名では聞かない）/ 市販なら**原材料表示を撮影**→Vision OCR | 「自家製ドレの材料は？」 | 確定 or unknown |

**アレルゲンは質問にしない（確認型も含めて）。** 「アレルゲンは〜で合ってる？」も禁止——店主は「魚＝出汁由来」を結び付けられない。店主が答えるのは「何を使ってるか（事実＝材料）」だけ。アレルゲンは知識ベース（材料マスタ `allergen_flag` / CASCADE）が**導出**し、最終チェックは公開ゲート（§5）で**表示のみ**。

代わりに聞くのは**材料軸の質問**で、店舗共通の回答で選択肢を絞る:
- 例（だし巻き玉子）: ✗「アレルゲンは卵・魚で合ってる？」 → ◯「この料理の出汁は？ 1.鰹/煮干し 2.昆布 3.椎茸 4.その他」→ 鰹なら魚を導出。
- 店舗共通で「お店の出汁＝鰹のみ」と答えてあれば、個別は「1.鰹 2.その他」まで簡略化、または継承で**質問ゼロ**。
- 構造化された材料軸の質問（出汁/衣/油…）は材料語彙とCASCADEを持つ **`kitchen_profile` が担当**。`question_engine` は材料が全く不明なときの汎用フォールバック（「主な材料を教えて／原材料表示を撮って」）だけ持つ。

> ⚠ 既知の穴: `kitchen_profile` の出汁CASCADEスコープは `{soup,nabe,steamed,rice,soba}` カテゴリ依存で、**だし巻き玉子(side)は「だし」と名前にあるのに対象から漏れる** → 魚が自動付与されない。個別の材料軸質問（or スコープをキーワードでも判定）で補う必要がある。

## 4. 状態モデル（生成 ⇄ 回答 ⇄ 書き戻し ⇄ 再質問抑制）

生成ルールだけでは回らない。回答後の状態を `field_confidence`(JSON列・**移行不要**) に持つ。

### field_confidence の規約（統一後）

```jsonc
{
  // 既存: フィールド別 確信度 0.0–1.0
  "allergens": 0.7, "ingredients": 0.5, "cooking_methods": 0.4, ...,
  // 店主/一次情報で確定したフィールド名（present/absent/complete を確定）
  "verified_fields": ["allergens", "dashi"],
  // 店主が「分からない」と確定した終端フィールド（再質問しない・客側に未確認表示）
  "unknown_fields": ["gelatin"],
  // 特定原材料の個別 tri-state（§5）
  "allergen_status": { "egg": "unknown", "wheat": "present", "milk": "absent" }
}
```

> ⚠ 既存の不整合を統一: `verified_fields`（confidence.py）と `store_verified`（kitchen_profile.py）の**2系統を `verified_fields` に一本化**。CASCADE は「kitchen_profile」という粗いタグでなく、**確定させたフィールド名**（例 `allergens`）を `verified_fields` に積む。→ これを直さないと店舗共通で埋めたアレルゲンを個別でも再質問する二重質問バグが残る。
> **Phase 1 では `store_verified` を信頼しない**。旧タグは「CASCADEが走った」を意味するだけで「そのフィールドを確認した」保証がないため。旧データの `verified_fields` への正規化は Phase 2 マイグレーションで行う。

### 回答 → 書き戻しの対応表

| 回答 | 書き戻し | 再質問 |
|---|---|---|
| 合っている / 直した | 値を更新 + `verified_fields` に追加 + rank昇格 | しない |
| 該当なし（無し確定） | 値を空に + `verified_fields` に追加（"verified empty"） | しない |
| わからない | `unknown_fields` に追加（AI推定値は残す/参考） | しない（終端） |

`verification_rank`: `C`(AI推定/未確認) → `B`(店主確認) → `A/S`(一次情報) に統一。
> ⚠ 既存不整合: `confidence.py` は `C` を「確定」、`kitchen_profile.py` は `C` を「未確認」と**逆に解釈**している。`C=未確認` に統一する。

## 5. アレルゲン特別ルール（安全最重要）

アレルゲンは「店主も知らない」が頻発し（仕入れソース・OEM・既製タレ）、かつ**"不明"を"無し"に丸めると事故**（賠償直撃）。だから特別扱い。

- **2値（present/absent）でなく tri-state**: `present` / `absent` / `unknown`（特定原材料8品単位）。
- **値が空の既定は `absent` でなく `unknown`**。空のとき「無しで合ってる？」と店主に否定を断定させない（マヨ/ドレ問題）。値ありは導出結果を確認型でレビュー、空は材料 or 原材料表示の写真で解決（L3）。`absent` は店主が明示的に否定したときだけ。
- 客向けAIの返答を状態で変える:
  - present → 「卵が入っています」
  - absent（店主確定）→ 「卵は使っていません」
  - **unknown → 「確認が取れていません。お店/スタッフにご確認ください」**（絶対に「入っていません」と言わせない）
- `confidence.py` の **「verified で allergens 空＝安全満点」を廃止**。unknown が残る料理は満点にしない。
- **公開ゲート**: unknown アレルゲンで publish を**ブロックしない**（OEM多用店が永遠に公開できなくなる）。**公開はさせるが客側に"未確認"を出す**。これが安全と実用の両立点。
- unknown を死に筋にしない: 「分からない」を選ぶと「**原材料表示の写真を送って**」→ 既存 `analyze_image`(Vision) が原材料欄をOCR → unknown を confirmed に昇格。

## 6. 5つの発火条件（質問を生む引き金）

| # | 引き金 | 形式 | 備考 |
|---|---|---|---|
| 1 | 安全項目で未確認 | 値あり=確認型 / 値なし=列挙 | safety は cap対象外 |
| 2 | 確信度が低い | §2の帯で決定 | askable フィールドのみ |
| 3 | 店舗共通と矛盾 | 確認型 | 例「このラーメンだけ豚骨？」。候補分布が未保存な今は名前/カテゴリのヒューリスティック止まり（別ジョブで強化） |
| 4 | 利用ログで頻出 | 確認型/選択 | 客質問N回(既定5)で発火。多言語分類が要る別ジョブ。トラフィック後にしか効かない |
| 5 | 日替わり新規 | **item_confirm**（§7） | 撮影直後 |

---

## 7. 日替わりメニュー（item_confirm）

フィールド単位でなく**料理アイデンティティまるごと**の確認タイプ `item_confirm`。

```
写真 → AI構造化 → 「本日の定食はこれで合ってる？（鯖味噌/小鉢/味噌汁/ご飯）」
   → 1.合っている / 2.修正する
   → YES: 店舗共通プロファイルを自動継承（CASCADE）→ 例外の安全項目だけ追加質問 → 公開
```

> ⚠ 継承の注意: 日替わりは「いつもと違う油・出汁」が起きやすい。**店舗共通の自動継承はデフォルトONだが上書き前提**。継承した値は `verified_fields` に積まず "AI継承" 扱い（後で安全項目だけ確認型で出す）。

## 8. 質問オブジェクト（チャネル非依存スキーマ）

```jsonc
{
  "key": "allergens:<menu_uid>",      // 重複排除の安定キー
  "menu_uid": "...", "menu_name": "鯖味噌定食",
  "tier": "safety",                    // safety | order | appeal
  "target_field": "allergens",
  "format": "confirm",                 // confirm | select | multi_select | text | item_confirm
  "question": "『鯖味噌定食』のアレルゲンは【小麦・卵】で合っていますか？",
  "ai_guess": ["wheat", "egg"],        // 確認型で見せる現在値（無ければ null）
  "options": [{"value":"egg","label":"卵"}, ...],  // 列挙/編集用（不要なら null）
  "actions": [{"value":"correct","label":"合っている"}, ...],  // 共通の出口(§2)。エンジンが必ず出力
  "allow_photo": true,                 // 原材料表示などの写真受付を許可するか
  "trigger": "safety_unverified",      // §6
  "priority": 100.0                    // 大きいほど先
}
```

LINE側はこれを「1メッセージ1質問・選択肢最大4・自由入力は最後の手段」に整形して配信するだけ。

## 9. 実装フェーズ

- **Phase 1（本仕様）**: `question_engine.py`（純ロジック・チャネル非依存・DB移行なし）。生成ルール+形式出し分け+ティア+5引き金。安全項目を完成、order/appeal は箱のある物だけ。
- **Phase 2**: 回答→書き戻しハンドラ（`verified_fields`/`unknown_fields`/`allergen_status` 保存）。`confidence.py` の verified空満点廃止 + rank `C` 統一 + `store_verified`→`verified_fields` 一本化。
- **Phase 3**: 公開ゲート（unknown→客側"未確認"表示）。客向けAIの unknown 応答変更。
- **Phase 4**: 日替わり item_confirm フロー + 原材料表示OCR昇格。
- **Phase 5**: LINE 配信アダプタ（既存 `feature/line-bot-phase1`）。
- **Phase 6**: Layer3 利用ログ分析ジョブ（多言語分類）+ ③矛盾検知の強化（候補分布の永続化）。

別タスク: アレルゲン導出知識を CASCADE_RULES のハードコードから材料/製品マスタ（`allergen_flag` / `product_master` 拡張）へ一元化。

## 補足: 材料 allergen_flag の運用ルール（2026-06-06 確定）

`ingredients.allergen_flag`（カンマ区切り小文字トークン）に材料→アレルゲンを持たせ、料理は材料から導出する（§3）。2026-06-06に頭〜中頻度 593語を投入（none明示247含む）。到達: アレルゲン検出68% / 全材料解決48%。

**鉄則: 「不明」を `none` にしない。**
- `none` = 「確認済み・アレルゲン無し」（"○○入ってません"と断言できる）
- 未分類(空) = 「不明」（→ 店主確認/写真OCRに回す）
- 例: `出汁/だし/つゆ` は昆布だし/精進だしの可能性があるので **fish にも none にもせず未分類**。鰹だしを `none` と断言すると魚アレルギー事故。名指しの魚だし(`煮干し/鰹だし/あごだし`)だけ `fish` 確定。
- 料理名から複合アレルゲンを推定しない（`ハンバーグ→beef`のみ、`カレー→未分類`）。卵乳小麦等は実材料から導出。

**実装済み(2026-06-06): allergen_flag に確信度を持たせる。**
- 形式: `allergen_flag` 文字列にトークン:確信度を埋める。**裸トークン=1.0(定義的)** / **`:数値`付き=蓋然的**。例 `egg`(=1.0) / `soy,wheat`(=1.0) / `fish:0.3` / `wheat:0.5,milk:0.4`。DBマイグレーション不要。
- 既存データは裸トークン=全て1.0 として後方互換。`kitchen_profile.allergen_has_ingredient_source` は `:確信度` を除去して比較。
- 導出 `allergen_derivation.derive_from_ingredients` は token→確信度(複数材料はmax)を返し、`split_confidence(hi=0.8)` で **confident(≥0.8:客に断言可)** と **hypotheses(<0.8:店主確認)** に分割。
- `question_engine.build_hypothesis_questions` が hypotheses を確認質問化(「『X』は小麦を含みますか？AIは可能性ありと判断」)。確定アレルゲンは聞かない。
- これで曖昧なだし系を「未分類」で落とさず `fish:0.3` として保持し、確認に回せる（"不明≠none"の精緻解）。投入例: `出汁→fish:0.3` / `和風だし→fish:0.4` / `カレー→wheat:0.5,milk:0.4` / `ハンバーグ→beef:0.7,egg:0.5,milk:0.4,wheat:0.5`。
- ⚠ **客向け consumer(Phase 3)は必ず確信度を尊重**: confident のみ断言、hypotheses は「可能性／要確認」表示。生フラグを present 扱いしない。

### ★governance（最重要・固定）
**allergen_prior（材料→アレルゲンの事前確率/class）は確定値ではない。** question_engine の「発火・優先順位・質問形式」を決めるための事前確率であって、**客向け回答には使わない**。客に出してよいのは、**店主確認後の `allergen_status`（present/absent/unknown）だけ**。生の prior を「含む」と断言することを禁止する。

### 確信度の表現: 数値でなく class（2026-06-06 方針転換）
数値(0.98等)は誰も測っていない偽の精度。OMISEAIの目的は確率計算でなく「店主に何を聞くか」なので、**class を一次表現**にする。class は **basis(なぜアレルゲンがあると判断したか)** で決まる:

| class | confidence帯(目安) | basis | 質問ポリシー |
|---|---|---|---|
| `direct` | 0.95–1.0 | ingredient_direct（語自体が原料: 卵/小麦/えび/醤油/パン粉） | 原則聞かない（safetyは初回のみ軽く確認可） |
| `high` | 0.8–0.95 | ingredient_high（マヨ/チーズ/バター） | 軽く確認 |
| `dish_prior` | 0.4–0.75 | dish_prior（ハンバーグ/カレー/グラタン） | 確認（選択 or 確認型） |
| `ambiguous` | 0.2–0.5 | ambiguous（出汁/つゆ/味噌汁/タレ） | 単体では聞かず、利用ログ等と合算して発火 |
| `none` | 0 | — | 聞かない（無し確定） |

- 格納: `allergen_flag` のトークンに class を付す。裸=`direct`、`token:high` / `token:dish` / `token:amb`、`none` は据え置き。例 `味噌汁 → soy,fish:amb`（味噌=direct、だし=ambiguous）。
- 将来: 店主確認の実データ（YES率）が溜まったら class→学習confidenceに格上げ。**今はclass、データが出たら数値**。
